## 2. 왜 IPPO가 아닌 MAPPO를 선택하는가

### 2-1. IPPO의 구조적 한계

IPPO(Independent PPO)는 각 로봇이 자신의 센서 관측만 보고 독립적으로 훈련된다.
다른 로봇은 "갑자기 나타나는 움직이는 장애물"로만 인식되며, **협력(Cooperation)이라는 개념 자체가 성립하지 않는다.**

결과적으로 IPPO는 어떤 상황에서도 충돌을 능동적으로 피할 수 없다.
S1(정면충돌)에서 우연히 100% 전원도달이 나온 것은 단순히 목표 방향으로 가면 서로 비켜지는 레이아웃 덕분이며, S2~S4처럼 실제로 양보가 필요한 상황에서는 100% 충돌이 발생했다.

### 2-2. MAPPO의 구조적 강점

MAPPO(Multi-Agent PPO)는 훈련 시 **Centralized Critic**이 모든 로봇의 상태를 전지적 시점으로 평가한다.

```
IPPO Critic:  obs_robot_i → value_i          (로봇 1대만 봄)
MAPPO Critic: obs_all_robots → value_team     (전체 팀 상태 봄)
```

Critic이 "네가 지금 멈추면 다른 로봇이 지나갈 수 있다"는 **글로벌 협력 전략**을 학습할 수 있는 유일한 구조다.
추론(배포) 시에는 Critic을 버리고 Actor만 사용하므로, Jetson 추론 비용은 IPPO와 동일하다.

### 2-3. MAPPO의 현재 실패 원인 — 구조 문제가 아님

현재 MAPPO가 100% 교착에 빠진 원인은 **보상 설계(Reward)의 문제**다.

```
충돌 페널티:    -150  → 한 번 부딪히면 파산 수준
정지 페널티:    -0.15 → 충돌보다 훨씬 작음
```

에이전트 입장에서 합리적 결론: **"차라리 가만히 있자"** (= Cowardly Local Optimum)

## 3. MAPPO 재학습 플랜

### 3-1. 보상 수치 수정 (`warehouse_marl_env.py`)

| 파라미터 | 현재 | 수정 후 | 이유 |
|---------|:----:|:-------:|-----|
| `rew_collision` | -150.0 | **-50.0** | 탐색 포기 방지. 충돌해도 목표 도달로 만회 가능하게. |
| `rew_stationary` | -0.15 | **-0.8** | 정지 전략이 충돌보다 더 아프게. 강제로 움직이게. |
| Potential 이동 보상 | 없음 | **추가** | 목표에 가까워질수록 +, 멀어지면 -. Sparse reward 탈출. |

### 3-2. Potential 이동 보상 추가 예시

```python
# _get_rewards() 내부에 추가
delta_goal = (self._prev_dist_goal - dist_goal).clamp(-0.5, 0.5)
nav_reward  = self.cfg.alpha * delta_goal * 5.0
self._prev_dist_goal = dist_goal.detach()
```

### 3-3. 예상 학습 곡선

```
0 ~ 1000 iter  : 충돌률 상승 (정상 — 움직이기 시작)
1000 ~ 2000 iter: 충돌+교착 줄타기
2000 iter ~    : 충돌률·교착률 동시 하락, 전원도달률 출현
```

### 3-4. 실행 명령

```bash
# 기존 체크포인트 버리고 from scratch
python training/multi_robot/train_multi_robot.py \
  --num_envs 128 --max_iter 5000 --algo mappo
```

## 4. 결론

| 항목 | IPPO | MAPPO |
|-----|:----:|:-----:|
| 협력 전략 학습 가능 | ✗ | ✓ |
| Jetson 추론 비용 | 동일 | 동일 |
| 현재 성능 | 내비게이션 ✓ / 충돌 회피 ✗ | 충돌 회피 ✓ / 내비게이션 ✗ |
| 개선 방향 | 구조적 한계 — 개선 불가 | 보상 재조정으로 해결 가능 |

**MAPPO + 보상 대수술이 유일한 100점짜리 선택.**
